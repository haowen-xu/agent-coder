# 项目需求讨论（V1）

## 1. 背景与目标

- 目标：建设一个可持续迭代的 `Go + WebUI` 系统骨架，用于后续功能快速开发。
- 后端技术：Go、Hertz、GORM、Viper、slog、errorx。
- 前端技术：Vue3、Pinia、Element Plus、Vite、pnpm。

## 2. 讨论范围

### In Scope（本轮确认）

- 产品定位与目标用户
- MVP 功能范围
- 数据存储策略（SQLite / PostgreSQL 的使用场景）
- API 基础规范
- 权限与认证基础方案
- 发布与运维最小要求

### Out of Scope（暂不讨论）

- 复杂多租户隔离细节
- 高并发压测优化细节
- 完整 BI/报表体系

## 3. 待确认问题清单

## Q1 产品定位

- 这是内部管理后台、对外 SaaS，还是混合模式？
- 主用户是谁（管理员、运营、普通用户、外部客户）？

## Q2 MVP 功能

- 第一阶段必须上线的 3~5 个功能是什么？
- 是否需要用户体系（注册/登录/角色）作为 P0？

## Q3 数据与环境

- 开发环境默认 SQLite，生产环境 PostgreSQL，是否确认？
- 是否需要数据库迁移机制（例如版本化迁移）在 MVP 就上线？

## Q4 API 与错误模型

- 接口风格是否统一 REST + JSON？
- 错误返回是否采用统一结构：`code/message/details/request_id`？

## Q5 鉴权与安全

- 是否采用 JWT（access/refresh）？
- 是否要求操作审计日志（谁在何时改了什么）？

## Q6 非功能目标

- 可用性目标（例如 99.9%）是否有硬性要求？
- 关键接口性能目标（例如 P95 < 200ms）是否需要本阶段定义？

## Q7 前端交互

- WebUI 是否仅桌面优先，还是需要移动端适配？
- 是否需要国际化（至少中英）？

## Q8 自动化与交付

- 当前默认使用 `scripts/run_codex_on_plan.py` 管理计划执行，是否继续沿用？
- 是否需要本地一键检查命令（后端测试 + 前端构建）作为提交流程门槛？

## 4. 建议的 MVP 基线（草案）

- 鉴权：JWT 登录/登出 + 基础角色（admin/user）
- 用户管理：用户列表、创建、启停
- 系统健康：`/healthz`、`/api/v1/meta`
- 前端：登录页 + 控制台首页 + 用户管理页
- 数据层：开发 SQLite，生产 PostgreSQL

## 5. 验收标准（草案）

- 后端：`go test ./...` 通过
- 前端：`pnpm build` 通过
- 核心流程：登录 -> 查看首页 -> 用户管理基础操作可用
- 文档：架构、需求、代码规范、计划执行文档齐备
